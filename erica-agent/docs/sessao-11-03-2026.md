## O que foi feito nesta sessão

### 1. Lista de bolões por loteria (`listar_jogos_loteria`)
**Problema:** Quando o cliente pedia para ver uma loteria (ex: "pode ver a mega?"), a Érica às vezes mostrava só 3 de 5 bolões disponíveis ou listava em texto livre (alucinando valores).

**Solução:**
- Criada nova tool `listar_jogos_loteria` no `tools.ts`
- O servidor monta a lista completa direto do banco e envia via Evolution API
- A IA é obrigada a chamar essa tool — PROIBIDO listar em texto livre
- Campo `loterias_listadas[]` adicionado na sessão para rastrear quais loterias já foram listadas

**Arquivos:** `src/agent/tools.ts`, `src/services/session.ts`, `src/agent/erica.ts`

---

### 2. Fluxo de múltiplas cotas (`detectarPedidoMultiplasCotas`)
**Problema:** Quando o cliente dizia "quero 3 cotas da mega e 2 da lotofacil", a Érica confirmava como se já estivesse comprado sem mostrar os bilhetes individuais.

**Solução:**
- Função `detectarPedidoMultiplasCotas()` detecta padrões do tipo "3 cotas da mega", "2 da lotofacil"
- Injeta instrução no prompt para mostrar a lista → cliente escolhe → confirma 1 por 1
- Fluxo documentado no `prompts.ts` na seção "Pedido de múltiplas cotas"

**Arquivos:** `src/agent/erica.ts`, `src/agent/prompts.ts`

---

### 3. Tool `limpar_carrinho`
**Problema:** Quando o cliente pedia para esvaziar o carrinho ("pode esvaziar meu carrinho?"), a Érica respondia em texto dizendo "infelizmente não é possível" por não ter a capacidade.

**Solução:**
- Nova tool `limpar_carrinho` que zera `boloes_confirmados`, `boloes_oferecidos`, `loterias_listadas`, `ultimo_bilhete_mostrado`, `dados_cliente` e reseta a fase para `venda`
- Detecção server-side antes de chamar a IA: se a mensagem contiver "esvaziar", "limpar carrinho", "cancela tudo" etc. → chama a tool diretamente sem depender da IA

**Arquivos:** `src/agent/tools.ts`, `src/services/session.ts`, `src/agent/erica.ts`

---

### 4. Tool `ir_para_fechamento` + detecção server-side de frases de encerramento
**Problema crítico:** A fase de `fechamento` nunca era atingida. A IA ficava presa no loop de upsell/downsell indefinidamente mesmo quando o cliente dizia "chega", "só isso", "pode fechar".

**Causa raiz:** A lógica antiga de transição para fechamento exigia que todos os bolões de todas as loterias fossem esgotados — com 23+ bolões disponíveis, isso era impossível na prática.

**Solução em 2 camadas:**

**Camada 1 — Tool para a IA:**
- Nova tool `ir_para_fechamento` que a IA chama quando detecta intenção de encerrar
- Muda a fase para `fechamento` e limpa `dados_cliente` para coleta sempre fresca
- Prompt atualizado para instruir a IA a chamar a tool imediatamente em vários gatilhos

**Camada 2 — Detecção server-side (fallback):**
- `Situação A`: servidor verifica lista de frases de encerramento antes de chamar a IA
  - "chega", "basta", "só isso", "mais nada", "não quero mais", "tá bom", "pode fechar", "quero fechar", "finali zar", "encerrar", etc.
  - Se detectado + carrinho tem itens + fase é venda/upsell/downsell → muda para fechamento e envia revisão do carrinho direto do banco
- `Situação B`: servidor verifica se a última mensagem da IA era uma pergunta de revisão ("Posso fechar?", "Quer revisar?") e o cliente respondeu com confirmação simples ("pode", "sim", "ok", "fecha")

**Revisão do carrinho enviada pelo servidor (não pela IA):**
- Ao entrar em fechamento, o servidor monta a revisão com valores exatos do banco e envia diretamente ao cliente
- Elimina alucinação de itens ou valores incorretos na revisão

**Arquivos:** `src/agent/tools.ts`, `src/services/session.ts`, `src/agent/erica.ts`, `src/agent/prompts.ts`

### 5. Correção da ordem de coleta de dados (Nome → Telefone → CPF)
**Problema:** A IA pedia CPF antes de coletar nome e telefone. Às vezes dizia "reserva finalizada" sem ter feito nada.

**Causa raiz identificada:**
- O system prompt mostra no cabeçalho `Nome: Marcus | Telefone: 5512991434039` (dados do contato WhatsApp)
- A IA interpretava esses dados como se já fossem os dados formalmente coletados para reserva
- Então pulava direto para pedir o CPF, ou às vezes assumia que tudo estava completo e dizia "reserva feita"

**Fix aplicado no `prompts.ts`:**
- Função `getRetomadaContexto` atualizada para receber `nome` e `telefone` como parâmetros
- No contexto de fechamento, agora exibe aviso explícito:
  - `"⚠️ ATENÇÃO: O nome 'Marcus' e o telefone '5512991434039' visíveis no cabeçalho são dados do contato WhatsApp — NÃO são os dados formalmente coletados para a reserva. Só vale o que está em 'Dados coletados' abaixo."`
- Mostra o próximo dado a pedir na ordem correta: Nome → WhatsApp → CPF
- PROIBIDO dizer "reserva feita", "pedido confirmado", "finalizado" — só o servidor confirma

**Fix no call site:**
- A chamada `getRetomadaContexto(sessao)` estava errada (TS error silencioso)
- Corrigida para `getRetomadaContexto(sessao, nome, telefone)`

**Arquivos:** `src/agent/prompts.ts`

## ⚠️ Problema em aberto: coleta Nome → Telefone → CPF ainda não validada

### Status
O fix foi deployado mas **não foi testado em produção** ao final desta sessão. O comportamento correto ainda precisa ser confirmado.

### O que foi implementado (lado servidor)
O arquivo `src/services/coleta-dados.ts` já existe e faz detecção heurística de nome/telefone/CPF nas mensagens do cliente. Armazena progressivamente na sessão. Quando os 3 dados estão completos e o CPF é válido, dispara `fazer_reserva` automaticamente.

### O que ainda pode estar falhando (hipóteses)
1. **A IA ainda pode ignorar o aviso do cabeçalho** e continuar tratando `Nome: Marcus` como dado coletado, dependendo de como o GPT-4o interpreta a instrução de aviso.
2. **A ordem pode ainda ser violada** se o `coleta-dados.ts` detectar o telefone do WhatsApp como "telefone coletado" antes de o cliente informar formalmente.
3. **O aviso de "PROIBIDO dizer reserva feita"** pode não ser suficiente — a IA pode continuar dizendo variações como "tudo certo", "está confirmado" antes de o servidor completar a reserva.
4. **CPF inválido:** o loop de rejeição de CPF (pedir só o CPF sem nome/telefone) foi implementado antes mas pode ter conflito com a nova lógica de aviso de cabeçalho.

### Próximos passos recomendados para resolver definitivamente
1. **Testar** o fluxo completo em produção: comprar bolão → "chega" → revisão → "pode" → aguardar IA pedir nome → informar nome → telefone → CPF → reserva
2. **Verificar logs** da sessão de teste para checar se a IA respeita a ordem e não usa o cabeçalho como dado coletado
3. Se a IA ainda usar o cabeçalho:
   - **Opção A:** Remover nome e telefone do cabeçalho do system prompt completamente — usar apenas na saudação inicial
   - **Opção B:** Renomear os campos no cabeçalho para algo mais neutro, ex: `Contato WhatsApp: Marcus | Número WA: 5512991434039`
4. Adicionar log server-side no `erica.ts` para imprimir o estado de `dados_cliente` a cada mensagem de fechamento — facilita debug
5. Se a alucinação de "reserva feita" persistir: adicionar validação hard no server antes de responder — se a resposta da IA contiver "reserva feita", "pedido confirmado", "finalizado" e a fase não for `aguardando_pagamento`, substituir por mensagem genérica de espera
